# client

## Purpose

CLI + library crate. Owns directory scanning, format-v3 snapshot sync orchestration, local cache metadata, predictive hydration, summaries, and the watcher. Agent and history operations delegate to `feanorfs-agent-core`. It transports content without interpreting or merging it. Serializable result types are shared by library callers and `--json`.

## Ownership

- Crates: `feanorfs-client` produces binary `feanorfs` (sync + `serve` hub + agents) and library `feanorfs_client`. Agent workspace logic lives in `feanorfs-agent-core` — this crate re-exports `Runtime`, `Workspace`, and thin `agent.rs` / `conflicts.rs` wrappers.
- Modules under `client/src/`:
  - `lib.rs` — public API re-export surface. `open_api_client` performs mandatory metadata migration plus direct/mDNS/relay selection; `open_relay_api_client` is the diagnostic relay-only variant and still migrates first. Add new public functions here; downstream Rust consumers depend on this list.
  - `api.rs` — HTTPS/HTTP + in-process hub transport (`Backend::Http` | `Backend::Local`). `ApiClient::from_config` / `open_for_workspace`; private hubs add the invite-pinned public CA to normal reqwest/Rustls verification.
  - `hub.rs` — thin re-export (`pub use feanorfs_agent_core::hub::*`); `LocalHub` lives in `feanorfs-agent-core`. The `LocalHub` type is a JSON/state wrapper with durable `hub_state.json` + blob storage, not an Axum router. `feanorfs serve` uses the `LocalHub` from agent-core directly.
  - `migrate_sqlite/` — one-time SQLite → JSON/files migration coordinator (`migrate_workspace_stores`), journal (`metadata-migration.json`), cache state machine (`cache.rs`), hub state machine (`hub.rs`), WAL/fingerprint/archive helpers (`journal.rs`). Runs automatically on every `open_client_db` / `open_api_client` call before opening the store. Produces `.migrated-v1.db` archives next to each original SQLite file.
  - `cli/serve.rs` — `feanorfs serve`, `--gc-only`, and interactive `serve recovery export|import|rotate`; recovery passphrases are terminal-only and server-library code owns cryptography/persistence.
  - `recovery.rs` + `cli/recovery.rs` — versioned Argon2id + XChaCha20-Poly1305 offline workspace-capability kits and the interactive `recovery export|import` surface. Import decrypts/validates before local writes and passes `WorkspaceInvite` in-process to `run_start`; hidden tray input is a bounded stdin line, never argv/environment.
  - `cli/hub_service.rs` — first-machine automatic private-hub lifecycle. Owns `~/.feanorfs/hub-data`, the credential-free per-user service worker, atomic `listen-port`, readiness/manual-hub detection, HTTP→TLS local migration, CA-bound portable invite export, and optional outbound opaque-tunnel workers loaded from private atomic `hub-data/relay.json`. It composes server-library TLS/auth/runtime primitives and never owns blob logic.
  - `cli/service.rs` — per-workspace automatic sync lifecycle. Uses `service-manager` 0.11 for macOS/Linux user services and Task Scheduler on Windows; hidden `service run <folder>` owns the supervised watcher process.
  - `cli/pair.rs` — client-to-client pairing. `fnp1` uses secret-free ephemeral mDNS on LAN; `fnp2` embeds a public relay URL/128-bit session plus an 80-bit client-only secret for outbound WSS rendezvous. Both run the same SPAKE2 + AEAD invite delivery/key confirmation and attempt/expiry limits. Hidden `pair --tray` emits only the ephemeral capability and TTL to the bundled tray's captured pipe.
  - `cli/tray.rs` — hidden tray adapters. Receiver-side `tray join <folder>` reads one bounded stdin pairing capability, validates it into a zeroizing `PairCode`, and delegates in-process to `run_start`; no capability enters argv/environment/logs.
  - `endpoint.rs` — pre-operation private-hub endpoint selection. Derives the managed `.local` hostname from the pinned public CA, tries authenticated direct and CA-correlated mDNS paths, then falls back to the configured inner-TLS relay without changing trust.
  - `cli/history.rs` — human and JSON `log` / `undo` output.
  - `cli/update.rs` — workspace-independent official stable-release lookup. Owns the HTTPS-only timeout/size bound, `semver` comparison, draft/prerelease rejection, exact GitHub tag-URL validation, typed human/JSON result, and the no-download/no-execute boundary used by the tray.
  - `commands.rs` — sync/push/pull/hydrate/cat/status via unified `run_sync_pass`. Owns `MirrorState` with `human_label()`. Returns `Serialize`-derived structs. No `println!` — UI in `main.rs` / `cli/`.
  - `conflicts.rs` — workspace conflict detection, registry, `resolve_conflict`, join/attach divergent-path guards, placeholder corruption, post-upload create/create, case conflicts. `seed_last_synced_from_server` skips same-path hash mismatches. `conflicts history` reads `conflict_resolutions`.
  - `conflict_artifacts.rs` — shared `.original`/`.local`/`.cloud` writer for agent and workspace conflicts.
  - `cli/` — CLI helpers (`util`, `agent`, `conflicts`, `serve`, `start`, `mcp`, `events`, `workspace`, …). Keeps `main.rs` under 1k lines.
  - `fs_util.rs` — `atomic_write` (temp + rename), `file_mtime_ms`.
  - `local.rs` — `Config` (`hub_local`, `format_version`), `ClientDb`, `scan_local_directory`.
  - `agent.rs` — re-exports `feanorfs_agent_core` agent ops; CLI `--json` uses the same shapes as [docs/agent-api.md](../docs/agent-api.md).
  - `predictive.rs` — `record_access_with_recent`, `prefetch_related` (top-5 siblings, 0.95 decay). Local-only.
  - `recent.rs` — locked, atomic `~/.feanorfs/recent.json` registry shared by `start`, `stop`, and the tray. Removing an active workspace deterministically selects the next entry.
  - `summary.rs` — `diff_since_last_session`, `commit_session_marker`, `render_via_summary_tool`. Zero-knowledge — never ships file contents to a remote LLM.
  - `watch.rs` — debounced (500 ms) filesystem watcher that drives `do_sync` on changes. Watcher path MUST be the workspace `current_dir`, never `"."`.
  - `tray.rs` — tray dashboard aggregation (`do_tray_status`, `build_conflict_show`). Agent summary cached on disk at `.feanorfs/tray-agent-cache.json` (30 s TTL); `invalidate_agent_cache` after land/keep.
- Local runtime data lives in `.feanorfs/` (git-ignored by FeanorFS itself; never include in distributions). Live state files: `.feanorfs/local_state.json` (format version, server URL, auth) and embedded hub `hub_state.json` + `blobs/<hash>` at `.feanorfs/hub-data/`.

## Local Contracts

- All paths stored in `local_state.json` use forward slashes via `feanorfs_common::normalize_path`. Always normalize before cache lookup or mutation.
- Legacy SQLite stores (`local_cache.db`, `agents/<name>/.feanorfs/local_cache.db`, `hub-data/db.sqlite`) are migrated one time to JSON/state files on first open. Migration is idempotent: re-running is a no-op once the journal shows `Archived` for all stores.
- Avoid redundant hashing: check `local_state.json` first and re-hash only if `mtime`/`size` differs from the cached entry. For unchanged placeholders (`hydrated=false`, `size==0`), reuse the cached hashes so the sync diff remains correct without downloading bytes.
- **Migration contracts:** lock (`.feanorfs/metadata-migration.lock`, `fs2` exclusive) gates the journal; journal (`.feanorfs/metadata-migration.json`) records store key, source paths, fingerprints, phase (`Discovered → Imported → Verified → Archived`), and archive paths; archives land next to each original SQLite as `.migrated-v1.db` (+ `-wal` + `-shm`) with hash-collision guard. Resuming after interruption reads the journal and skips completed phases.
- **Sync scope:** mirror the working directory (including gitignored/untracked paths). Hard skip `.feanorfs/`, `.git/`, symlinks, and nested directories declaring a valid `CACHEDIR.TAG` (workspace-root tags are exempt). Small frozen `DEFAULT_IGNORES` plus optional `.feanorfsignore` — does NOT honor `.gitignore`. Rationale and admission criteria: [docs/sync-scope.md](../docs/sync-scope.md). Do not grow `DEFAULT_IGNORES` without meeting all three criteria there.
- Zero-knowledge: seal file bytes before transport and verify ciphertext hashes before decrypting. Format-v3 uploads file blobs through opaque object storage and commits encrypted trees through head compare-and-swap.
- Result types are `Serialize`-derived. The `--json` CLI flag and `feanorfs_client::` library callers MUST see the same shape; do not add `println!` in `commands.rs` or `agent.rs` — keep UI in `main.rs` only.
- MCP `sync_status` is intentionally a compact projection of `StatusResult`: preserve mirror state, local-file count, actionable pending paths, conflict/offline state, rollback warning, and skipped-symlink count, but never serialize the complete `local_files` map into routine MCP responses. CLI `--json status` and library callers retain the full result.
- Human `status` output shows at most five deterministic symlink examples plus the total; `StatusResult.skipped_symlinks` and `--json` retain the complete sorted list for automation.
- Workspace conflicts: bare `feanorfs conflicts` or `conflicts list`; `keep`; `show [--open]`. Registry in `conflict_registry`; artifacts under `.feanorfs/conflicts/<ts>/`. `ResolveKeep::Cloud` on `edit_delete` conflicts removes the local file and uploads a tombstone when the cloud artifact is a deletion sentinel.
- `agent status` (or bare `agent`) lists agents with one-line state when online; hidden `agent list` keeps legacy JSON shape. `agent status <name>` / hidden `agent check` peek via `/api/sync/peek`.
- Agent workspaces isolate DATA, not processes — `agent run` is cwd-scoping only. Never claim sandboxing; link [docs/threat-model.md](../docs/threat-model.md) § Process isolation.
- Predictive hydration is local-only: `file_access_log` never leaves the client.
- Local-hub workspaces (`hub_local` / `start --local`): in-process transport via `LocalHub` (agent-core JSON/state at `hub_data_dir/hub_state.json` + `blobs/`). `hub.rs` is a re-export shim; the Axum router for `feanorfs serve` is in `cli/serve.rs`. Do not print portable invites; share with `feanorfs serve --data-dir .feanorfs/hub-data`.
- Sync reconciliation compares local and head trees with `.feanorfs/refs/last-synced`. The `last_synced_files` and `agent_snapshots` tables are removed in format v3. `mtime` remains cache and rollback evidence, never content identity.
- Format-v3 migration journals old key, target key, fence token, and phase in `.feanorfs/migration-v3.json`; this is distinct from SQLite import journal `.feanorfs/metadata-migration.json`. Rekey never persists the target key before reseal, parentless head CAS, server stamp, and local finalization complete. Resume the command after interruption.
- Rekey requires clean or landed agent workspaces because old-key agent base refs cannot cross the key boundary.
- Onboarding: `start [target] [folder]` accepts an `fnh1-…` secure hub invite, full `fnr1-…` workspace invite, single-use LAN `fnp1-…` or off-LAN `fnp2-…` pairing capability, or folder. With no workspace or cached connection, a folder target creates the automatic secure private hub; `--host` explicitly selects that path and `--relay <URL>` adds outbound-only off-LAN reachability without a new command. Every branch converges on link/create → initial sync → workspace service + tray. Hidden `tray join <folder>` is the bundled receiver UI adapter: it accepts only bounded stdin `fnp1`/`fnp2`, constructs the existing zeroizing pairing type, and enters `run_start`. Hidden `attach`/`init`/`setup` configure only.
- Positional target detection must preserve Windows drive paths and ordinary colon-bearing folder names. Scheme-free server targets are accepted only when they are unambiguous `localhost`, IP, or dotted-host values with an explicit port; custom single-label hosts require `https://`.
- On an already configured folder, `start fnh1-…` is a hub-trust refresh: require HTTPS plus a bearer token, authenticate the candidate CA/token and existing opaque head before writing config, and preserve workspace ID, E2EE key, format, refs, files, and history. A failed probe must leave old connection settings untouched.
- New mirrors without an explicit `--workspace` receive independent opaque random `fsw1-…` IDs. Never restore a shared `default` consumer ID: adding a second folder must not attach it to another mirror. Invites and pair codes preserve their existing workspace ID.
- New/link format-v2/v3 paths validate a manual E2EE key before writing workspace or global configuration. Accept only the canonical 64-character lowercase-hex recovery-key shape; automatic setup generates it. Legacy format-v1 keys remain loadable for migration, and `migrate --rekey` is the path for historical human-chosen keys.
- Workspace recovery kits contain the complete portable capability only as authenticated ciphertext. Use Argon2id v19 (64 MiB, three iterations, one lane), XChaCha20-Poly1305, a distinct authenticated domain, bounded input, atomic `0600` Unix writes, explicit overwrite, and fail-before-write import. The kit is not a blob backup. Decrypted invites remain in-process and converge on `run_start`; passphrases use terminal input or the bundled tray's bounded stdin pipe and never argv/environment/logs.
- Never use an accept-invalid-certificate client. Private hub CAs come from `fnh1`/`fnr1` capabilities; unauthenticated mDNS fingerprints are informational and must not create trust-on-first-use.
- CLI surface: visible verbs include `log` and `undo` alongside sync, workspace, agent, and conflict commands. Hidden tray commands continue to feed `feanorfs-tray`.
- `conflicts keep --all --local` is the bounded authoritative-folder path for
  resolving every pending conflict without one scan and snapshot publication
  per file. Other `--all` policies remain rejected until their distinct
  materialization semantics are implemented and confirmed.
- `update` is advisory release awareness, never an installer: make one manual bounded HTTPS-only, no-redirect lookup of the versioned official stable GitHub release API, parse with the direct maintained `semver` dependency, and return one typed status (`up_to_date`, `update_available`, `development_build`). Reject draft/prerelease/malformed metadata and any URL other than the exact matching official tag page. Never download, stage, install, execute, or silently open an artifact; signed/checksummed platform installers remain authoritative.
- `start` performs its initial sync, then installs/starts a per-user workspace service and returns. First-machine hosting installs the global hub service before creating the workspace. `--foreground` retains the terminal watcher and `--no-watch` remains one-shot for the workspace (the hosted hub still persists). Service metadata contains only canonical workspace/hub-data paths; keys/tokens stay in protected files or the OS store.
- `start` protects new and legacy local credentials through agent-core's OS-store facade before background execution. Embedded hubs never inherit a cached token from an unrelated remote hub.
- `service install|status|start|stop|uninstall [folder]` is the recovery surface. The tray is resolved from an explicit override, a colocated desktop product, `/Applications/FeanorFS.app` on macOS, or `PATH`, then registered through the per-user login manager; the workspace service remains the sync owner. Windows hub, workspace, and tray status use the stable Task Scheduler state enum, not localized text or task existence alone. Register only the tray with `InteractiveToken`; hub/workspace workers remain non-interactive.
- Windows task registration must store the executable and arguments as separate Task Scheduler fields; never flatten them through `schtasks /TR`, whose 261-character limit rejects ordinary long install/workspace paths. Task actions still contain only the canonical workspace/hub path and no credentials or capabilities.
- Tray `watching` state and sync-lock checks must reuse agent-core's native cross-platform process-liveness probe; a live Windows watcher cannot be reported dead or have its lock treated as stale.
- Consumer `stop [folder]` composes service uninstall with locked/atomic recent-workspace removal. It refuses unmanaged active watchers and preserves `.feanorfs`, working files, credentials, remote snapshots, and any private or embedded hub; rerunning `start` resumes the same workspace.
- Hidden `tray forget-unavailable` is the tray's explicit stale-list repair. Under the existing recent-store lock, remove only entries whose `.feanorfs/config.json` is not currently a file, select the next available active entry, and atomically replace `recent.json`. Never touch workspace files, credentials, services, hubs, or remote snapshots; the tray must confirm first because an external drive may only be disconnected.
- Hidden `service hub-run <data-dir>` is only the OS-supervised worker. Do not expose hub lifecycle vocabulary in normal onboarding. `start --host` safely converts a matching local legacy HTTP hub to native TLS while preserving its token; it refuses an active manual process and unrelated workspace identities.
- `doctor` accumulates secret-free named checks for global/workspace config, E2EE, format-v3 trees, automatic workspace sync, tray registration, private-hub persistence, authenticated endpoint reachability, remote workspace evidence, and local state. Human and `--json` output share the same result; one failing check must not prevent later checks from running. A manually reachable but non-persistent workspace or hub never passes.
- `doctor --migration-report` is a separate local evidence mode: read only the minimal `format_version` field from current/recent workspace config files, never resolve credentials or contact a hub, deduplicate folders, and emit a versioned aggregate schema only. Do not include paths, workspace IDs, labels, endpoints, credential references, routes, keys, tokens, or capabilities. Lock the exact JSON key set in tests. An unreadable recent registry, missing/malformed config, unsupported format, empty profile, or any format-v1 workspace must prevent a retirement-ready result; this local signal never authorizes removing legacy decryption without broader field evidence.
- A fresh automatic host prefers `https://127.0.0.1:3030`; if the port is occupied, it scans the bounded stable range 3030–3130 before requesting an OS-assigned last resort, then atomically persists the nonzero result as `0600` `hub-data/listen-port`. Existing hubs without that record retain 3030. The worker reads the validated port from the protected data directory, while service argv remains only `service hub-run <data-dir>`. TLS readiness, the relay tunnel, mDNS, diagnostics, and portable invites must use the persisted port. Exported full invites rewrite only loopback to the stable CA-derived `.local` hostname after the invite public CA matches `hub-data/tls/ca-cert.pem`; the port is preserved. The hub auto-tracks interface addresses. Receiving clients retain normal TLS verification, and legacy numeric URLs migrate only after a bearer-authenticated CA-verified probe.
- Private-hub disaster recovery is an advanced `serve recovery export|import|rotate` surface, not part of onboarding. It prompts for the bundle passphrase, passes no secret through argv/env, and delegates identity validation, encryption, locking, fencing, persistence, and fresh CA/token generation to `feanorfs-server`. Rotation preserves opaque storage but requires every existing client to authenticate the replacement `fnh1` capability.
- On macOS, `service stop` unloads but preserves the launchd plist so `KeepAlive` cannot immediately respawn the watcher, then waits for the terminating process to release `.feanorfs/sync.lock`; `service start` loads that preserved plist. Keep this behavior aligned with tray-exclusive actions.
- Automatic hub, workspace, and desktop tray jobs persist path-plus-Blake3 executable identities (`service-program` markers) and reinstall when bytes change even at the same package path. Resuming any locally owned workspace also checks/restarts its private hub; `--host` is not required for package upgrades. Background `start` temporarily stops an existing managed watcher before its initial sync, restores it on failure, and drops foreground workspace handles before launching the replacement worker, so upgrades neither race the sync lock nor leave old mapped binaries running.
- `pair` is client-layer introduction, not a hub endpoint. Managed hubs deliver their CA-bound stable hostname; custom loopback hubs retain the accepted-interface IP fallback. The `fnp1` secret never appears in mDNS, tray receiver argv/environment/logs, or hub storage; the full invite is SPAKE2-authenticated, AEAD-encrypted, key-confirmed, and zeroized where owned. Hidden tray presentation may emit only the ephemeral code and TTL through captured stdout and must stop the child when its native dialog closes. Receiver-side tray input is masked and bounded, uses the same `PairCode` and `run_start`, and never duplicates validation or exchange logic. Preserve full `fnr1` as the non-LAN fallback.
- `fnp2` off-LAN pairing requires WSS except loopback tests, embeds the relay URL and 128-bit public session in the zeroized capability, keeps the 80-bit secret out of relay URLs/state/logs, and reuses SPAKE2/AEAD/key confirmation unchanged. Public HTTPS hubs need only rendezvous; owned private `.local` hubs first persist a validated 256-bit tunnel route through `start/pair --relay`, restart the hub service transactionally, and include that route in the encrypted invite. The tray reuses stored relay settings automatically.
- Relay-backed sync must preserve the original HTTPS hostname as SNI, pinned/system CA verification, and bearer auth. Direct/mDNS paths are attempted first by the CLI; relay fallback is bounded and never places the route, token, or invite in service argv/environment/logs. The hub worker reads the route only from atomic `0600` `hub-data/relay.json`, so it does not depend on ambient `HOME`. `doctor` performs a real relay-only authenticated probe when configured.

## Work Guidance

- New public functions go into the appropriate module and are re-exported from `client/src/lib.rs`.
- Keep service commands idempotent and per-user. A normal `start` must never require root/Administrator privileges, `nohup`, PID files, or credentials in argv.
- Automatic-hub changes must test secure defaults, worker argv, collision fallback, atomic/private port persistence, existing-hub compatibility, malformed-port failure, manual legacy-port refusal, matching-identity migration, login-service restart, TLS 401 without auth, authenticated client readiness, and portable invite URL rewriting.
- Pairing changes require correct-code, wrong-code, frame-bound, expiry/attempt, loopback URL rewrite, and real LAN discovery verification. Never log the pairing code, invite, server token, or E2EE key.
- `commands::password_or_default` warns when falling back to `LEGACY_DEFAULT_PASSWORD`. Treat any codepath that needs the default as a bug.
- Summary JSON shape (`SummaryResult`) is consumed by `FEANORFS_SUMMARY_CMD` via stdin — coordinate before renaming fields.
- After ANY code change in `commands.rs` or `local.rs`, run `cargo clippy --workspace --all-targets -- -D warnings` and `cargo test --workspace`.
